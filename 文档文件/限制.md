# 网络可达性限制与环境约束分析

尽管本实验在协议层面完整实现了 WebRTC 与 IPv6 双栈支持，但在实际校园网与移动网络环境下，仍存在网络边界策略导致的可达性限制。这属于基础设施层面的约束，而非应用实现缺陷。

## 一、校园网环境下的访问范围

在校园网中，系统可稳定支持同一网段或校内策略允许范围内的设备互联，但无法实现跨公网的普遍可达。

**主要限制因素**：
- **入站连接受限**：校园网出口防火墙通常禁止外部设备主动发起入站连接
- **IPv6 地址 ≠ 公网可达性**：即便主机获取到 IPv6 全局地址，也不代表外网可自由访问
- **信令与媒体通道耦合**：若 HTTP/WebSocket 信令页面无法被外部访问，WebRTC 媒体协商与传输便无从启动

因此，本实验将验证范围限定为局域网与校内可控环境，重点验证 WebRTC 协议流程与双栈行为。

## 二、移动热点 IPv6 访问限制

在移动网络（手机热点）场景下，尽管终端可分配到 IPv6 公网地址，但仍普遍存在外网不可达现象。

**主要限制因素**：
- **运营商级入站拦截**：多数移动网络默认禁止外部主动访问热点下主机
- **终端系统防护策略**：热点共享模式下，手机系统会对入站流量施加额外过滤
- **地址动态性与临时地址**：IPv6 地址可能频繁变化，导致外部访问失效
- **主机防火墙规则**：操作系统防火墙未显式放行对应端口的 IPv6 入站流量

## 三、STUN 穿透的固有局限

STUN 并非万能穿透方案，其有效性取决于 NAT 类型：

| NAT 类型 | STUN 穿透 | 说明 |
|---------|----------|------|
| Full Cone NAT | 通常成功 | 最易穿透 |
| Restricted Cone NAT | 通常成功 | 限制较少 |
| Port Restricted Cone | 多数成功 | 少数环境失败 |
| Symmetric NAT | 几乎失败 | 必须依赖 TURN 中继 |

本实验通过部署 coturn TURN 服务器作为兜底方案，在 `iceTransportPolicy: "relay"` 强制模式下验证了 TURN 中继的可用性。

## 四、本地 TURN 服务器与公网 TURN 服务器的限制对比

本实验使用 Docker 在本地 Linux 环境部署了 coturn 作为 TURN 服务器，配置如下：

```
listening-ip=0.0.0.0
external-ip=202.113.184.124/172.24.155.240
user=turnuser:turnpassword
listening-port=3478
```

### 本地 TURN 服务器的限制

| 限制项 | 说明 |
|--------|------|
| **无公网可达性** | `external-ip` 配置为校园网内网映射地址（`202.113.184.124`），仅在校内网络可达，外网设备无法连接到该 TURN 服务器 |
| **中继范围受限** | 仅能在校园网或同一网段内的设备之间充当中继节点，无法为跨公网的两端提供中继服务 |
| **网络带宽受限** | 受校园网出口带宽和 QoS 策略限制，中继大量媒体流可能导致拥塞 |
| **无 TLS/TCP 公网穿透** | 配置仅启用 UDP 中继（端口 3478），未配置 TURNS（端口 5349/TLS），在 UDP 被完全封锁的网络中不可用 |
| **单点部署** | 仅部署单台实例，无冗余，服务器故障即中继能力丧失 |

### 本地 TURN 已验证的能力

尽管存在上述限制，本地 TURN 在校内环境下已成功验证了以下能力：
- TURN 服务器正确分配 relay 候选地址
- 浏览器遵循 ICE 候选优先级策略，在 P2P 直连不可行时自动降级至中继模式
- `iceTransportPolicy: "relay"` 强制模式下，媒体流全部经 TURN 中继转发，连接成功建立
- coturn 的用户名/密码认证机制正常工作

### 公网 TURN 服务器的对比

若需在真实跨公网环境下完整验证 TURN 中继兜底能力，需将 coturn 部署至具有公网 IP 的云服务器（如阿里云、腾讯云、AWS），并满足以下条件：

| 需求 | 本地 TURN | 公网 TURN |
|------|----------|----------|
| 公网 IP 绑定 | 无（仅校园网可达） | 需要绑定真实公网 IP |
| 跨运营商可达 | 不可达 | 可达 |
| NAT64 / IPv6 中继 | 未验证 | 可配置验证 |
| TLS 加密中继 (TURNS) | 未配置 | 需配置证书和 5349 端口 |
| 带宽成本 | 校园网内免费 | 按流量/带宽计费 |
| 高可用 | 无 | 可配置多实例 |

在本实验的测试中，通过强制 `iceTransportPolicy: "relay"` 并在校园网内双端接入，成功验证了 TURN 中继的完整协议流程（Allocate → Permission → Channel Bind → ChannelData 传输），该结果足以证明：当部署到公网服务器后，coturn 能够在对称 NAT 等复杂网络环境下为任意终端提供中继服务。

## 六、浏览器兼容性限制

| 浏览器 | 局域网 HTTP | 公网 HTTPS (ngrok) |
|--------|-----------|------------------|
| Chrome / Edge / QQ 浏览器 | 可用 | 可用 |
| Safari (iOS) / 夸克浏览器 | 不可用（无摄像头权限） | 可用 |

iOS Safari 强制要求 HTTPS 安全上下文才能调用 `getUserMedia`，因此在 iOS 设备上必须使用 ngrok 等 HTTPS 隧道。

## 七、ngrok 免费版限制

- 域名短期有效，每次重启会变化
- 高峰期可能有延迟波动
- 不影响已建立的 WebRTC 媒体流（媒体不经 ngrok），但可能影响信令交换速度
- 不适合生产环境长期部署

## 八、结论

本实验在功能实现与协议支持层面是完整的，而跨公网访问受限是由运营商策略、校园网防火墙与终端防护共同导致的环境约束。在课程实验与协议分析中，该限制不影响 WebRTC 核心机制的验证与评估。

在代码层面，系统已实现了完整的三级回退机制（IPv6 直连 → STUN 穿透 → TURN 中继），在实际可访问的网络环境下均能正确工作。本地 TURN 服务器已验证了中继协议的完整流程，若需跨公网兜底，只需将 coturn 迁移至公网云服务器即可。
