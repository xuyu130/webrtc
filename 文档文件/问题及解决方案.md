# 开发过程问题记录

## 网络环境相关

### Q1：校园网环境下，外网无法访问本机部署的前端页面

**问题**：校园网环境下公网 IP 无法直接访问本机服务，导致跨网络环境的信令建立阶段就无法进行。

**解决方案**：使用 ngrok 内网穿透工具，将本地服务映射为公网 HTTPS 地址。ngrok 仅负责信令通道（WebSocket）的公网映射，WebRTC 媒体流仍为 P2P 直连，不受影响。

### Q2：手机热点获取 IPv6 公网地址后，外网仍无法访问

**原因分析**：
1. 运营商默认拦截入站连接：移动网络提供 IPv6 地址，但仅允许出站，不允许外部主动入站
2. 手机系统入站防护：热点共享模式下对入站流量施加额外过滤
3. IPv6 地址动态性：地址可能频繁变化或使用临时地址
4. 本机防火墙未放行对应端口的 IPv6 入站流量

**结论**：移动网络虽提供 IPv6 地址，但受运营商入站访问策略与终端防火墙影响，公网设备无法直接访问热点下主机服务，属于网络环境约束而非应用实现缺陷。

## 技术实现相关

### Q3：IPv6 直连是否一定能成功？

**解答**：不一定。需要同时满足三个条件：
1. 浏览器确实采集到了 IPv6 host 候选并通过信令发出
2. 对端实际可达该 IPv6（非仅链路本地或运营商不可路由）
3. 双方的 UDP/防火墙允许 IPv6 UDP 数据包

任一环节失败都会导致无法传输，系统会自动回退到 STUN 穿透或 TURN 中继。

### Q4：为什么同时绑定 0.0.0.0 和 ::？

**解答**：`0.0.0.0` 是 IPv4 通配符，`::` 是 IPv6 通配符。在某些系统上仅绑定 `::` 可能同时接受 IPv4 和 IPv4-mapped IPv6 连接，但行为不可靠。显式同时绑定两者可确保双栈兼容性，对应 RFC 3493 的 IPv6 Socket API 规范。

### Q5：ngrok 免费版的限制

- 域名短期有效，每次重启会变化
- 带宽受限
- 不适合长期在线
- 但足够用于实验验证和演示

## 系统运行相关

### Q6：WSL 中启动服务，Windows 浏览器无法访问

**原因**：WSL 有独立的网络命名空间，Windows 不能直接通过 `localhost` 访问 WSL 中的服务。

**解决方案**：
- 在 Windows 主机环境启动服务（推荐）
- 或配置 WSL 端口转发

### Q7：coturn 配置要点

- `external-ip`：coturn 所在主机的公网 IP / 内网 IP 映射
- `user=turnuser:turnpassword`：静态凭据
- `listening-port=3478`：标准 TURN 端口
- `fingerprint` 和 `lt-cred-mech`：启用安全校验
- Docker 使用 `network_mode: "host"`（仅 Linux 支持）
