# 系统测试方案与验证结果

## 一、测试体系设计

为全面验证"C/S 信令 + P2P 媒体"混合架构在 IPv4 / IPv6 双栈环境下的功能正确性、协议兼容性与传输鲁棒性，本章设计了四维测试体系。

### 测试维度

| 测试维度 | 核心指标 | 验证目标 |
|---------|---------|---------|
| 功能性测试 | 视频传输成功率 | WebRTC 连接建立、信令交换、媒体回退 |
| 协议性测试 | IPv4/IPv6 候选占比 | 双栈自适应、NAT 穿透、TURN 兜底 |
| 性能性测试 | 帧率 / 码率 / 延迟 | 实时性、网络自适应、拥塞控制 |
| 可靠性测试 | ICE Restart 成功率 | 弱网恢复、连接自愈能力 |

---

## 二、详细测试场景

### 场景 1：基础功能测试（本机 IPv4）

**目的**：验证基本的 WebRTC 连接和视频传输。

**验证内容**（基于第 5 章基础验证）：
- 信令通道（WebSocket）稳定连接
- SDP Offer/Answer 正常交换
- 本地与远端视频实时渲染
- 前端 UI 状态与协议栈同步
- IPv6 地址发现与双栈监听

**验证结果**：全部通过。发送端和接收端加入同一房间后，信令状态由"未连接"变为"已连接（WebSocket）"，ICE 状态由"未开始"变为"正在收集（connecting）"，信令通道成功建立。

---

### 场景 2：STUN NAT 穿透测试

**目的**：验证 STUN 在 NAT 环境下的公网地址映射能力。

**双端配置**：
- PC 端：校园网
- 移动端：5G 流量
- 共同通过 ngrok 公网地址接入

**验证结果**：

1. **信令与 ICE 连接**：两端通过 ngrok 公网地址接入后，信令连接正常建立，双端界面显示 PeerConnection 和 ICE 连接状态均为 `connected`，运行日志中出现 STUN 协议交互记录。

2. **服务端日志**：出现 `[STUN]` 条目和本地候选地址记录，显示通过 STUN 协议，两端设备均成功获取到各自的公网映射地址。

3. **webrtc-internals 候选对**：
   - `local-candidate`：`candidateType=host`
   - `remote-candidate`：`candidateType=srflx`
   - `candidate-pair` 状态：`state=succeeded`
   - ICE 引擎选定 host-srflx 候选对作为传输路径，未选取 relay 中继，验证了低延迟特性。

4. **Wireshark 抓包**：捕获大量 STUN 协议数据包，包括 Binding Request、Binding Success Response 等，从底层网络通信角度验证了 STUN 在 NAT 穿透中完成地址映射和连通性检测的关键步骤。

5. **IPv6 协议支持**：通过报文详情，系统全程支持 IPv6，并在 IPv6 网络中通过 STUN 成功获取公网映射地址。

---

### 场景 3：TURN 中继兜底测试

**目的**：验证对称 NAT 或防火墙阻断场景下的 TURN 连接可用性。

**双端配置**：同场景 2。

**特殊配置**：强制 `iceTransportPolicy: "relay"`，屏蔽 HOST 与 STUN，仅通过 TURN 传输。

**coturn 配置**：通过 Docker 在本地 Linux 环境部署 coturn，开启 `external-ip` 公网地址映射、用户名/密码认证、IPv4/IPv6 双栈监听。

**验证结果**：

1. **服务端日志**：显示 `[TURN][room=demo][role=viewer] relay candidate (TURN): 202.113.184.141:59251`，TURN 服务器成功分配中继端口。

2. **webrtc-internals 候选对**：
   - `candidate-pair` 状态：`succeeded`
   - 双端 `candidateType` 均为 `relay`
   - IP 地址均为 TURN 服务器地址 `202.113.184.141`，端口分别为 59251（远端）和 59555（本端）
   - `relayProtocol` 为 `udp`

3. **Trickle ICE 验证**：候选者列表中 `relay` 类型候选者优先级最低（`3|34|255` 和 `1|33|255`），但在 P2P 直连失败后被选中，符合 WebRTC 候选优先级策略。

4. **Wireshark 抓包分析**（分为两个阶段）：

   **阶段一：TURN 中继资源分配与通道准备**
   - 客户端发送 STUN Binding 请求探测公网映射地址
   - 客户端向 TURN 服务器发起 Allocate 请求，服务器返回 401 未授权（标准安全机制）
   - 客户端携带正确凭据重新请求，获得 Allocate Success Response
   - 分配公网中继地址 `XOR-RELAYED-ADDRESS: 202.113.185.121:50779`
   - 客户端发起 CreatePermission 为对端地址申请通行权限，全部确认成功

   **阶段二：中继数据通道激活与媒体传输**
   - 客户端发起 Channel Bind 操作，将 UDP 通道号与中继地址绑定
   - 绑定成功后密集收发 ChannelData TURN Message（封装的 RTP/RTCP 媒体载荷）
   - TURN 服务器解封装并转发至对端，形成双向媒体流传输路径

---

### 场景 4：ICE Restart 弱网恢复测试

**目的**：验证网络中断后的连接自愈能力。

**双端配置**：PC 端校园网，移动端 5G，通过 ngrok 公网地址接入。

**初始状态**：连接为 `connected`，已通过 STUN 获取 IPv6 公网地址（`2403::...:49299`）与 IPv4 公网地址（`172.24.144.1:49298`）。

**操作**：PC 端断开并重新打开无线网。

**验证结果**：

1. **前端日志**：连接状态先变为 `disconnected`，接着触发 ICE disconnected，系统尝试 ICE restart，随后短时间内将连接恢复为 `connected`，重新输出远端画面。

2. **webrtc-internals 验证**：
   - ICE 连接状态跳变：`new → checking → connected → disconnected → connected`
   - 信令状态同步变化：`new → have-remote-offer → stable → have-local-offer → stable`
   - ICE Candidate pair 出现新候选对（ID: 49299→53612），表明 ICE Restart 已被触发并完成
   - 从 disconnected 到再次 connected 的间隔极短，验证了 ICE Restart 机制的高效性与鲁棒性

---

### 场景 5：媒体多源回退测试

**目的**：验证无摄像头环境下的健壮性。

**验证结果**：

1. **Canvas 模拟流回退**：在无物理摄像头条件下，系统自动切换至 Canvas 模拟流作为媒体源，成功建立 WebRTC 连接，跨端稳定传输与解码播放。全程无异常中断、无致命错误。

2. **本地视频文件推流**：手动选取本地 MP4 文件后，本地画面正常播放，远端画面正常显示，帧率信息正常呈现，未触发错误或断连。MP4 文件通过 `captureStream()` 转为视频流后经 WebRTC 传输。

---

### 场景 6：多房间并发隔离测试

**目的**：验证信令服务的并发处理能力。

**验证结果**：
- 房间"1"中跨主机（`127.0.0.1` 与 `192.168.56.1`）的发送端与接收端成功建立连接，信令、PeerConnection 和 ICE 均显示 `connected`
- 房间"2"的单机双标签页测试中，系统在无摄像头、无本地视频文件条件下仍能稳定完成多客户端并发接入与隔离
- 信令服务按房间号正确路由、互不干扰，资源占用轻量可控

---

### 场景 7：跨协议互通测试（IPv4 ↔ IPv6）

**目的**：验证双栈信令下，异构网络节点的互通能力。

**双端配置**：
- PC 端：VirtualBox 虚拟网卡，IPv6 地址（`fe80::...`）不具备路由能力，等效于"不含可用 IPv6 路径"
- 移动端：5G 网络（IPv6）

**验证结果**：
- 本端上报 IPv4 类型本地主机候选者（`172.24.144.1:49298`）
- 对端上报 IPv6 类型 STUN 映射候选者（`2408:8411:60b1:e441:e996:fee4:5a80:1fca:53282`，中国电信公网 IPv6）
- ICE 协商成功建立连接，状态最终为 `connected`，曾因网络中断触发 ICE restart 并快速恢复
- 说明系统在双栈信令环境下实现了 IPv4 与 IPv6 异构网络节点间的媒体互通，验证了 STUN 服务的双栈解析能力

---

### 场景 8：IPv6 纯网直连测试

**目的**：验证在双栈网络环境下，WebRTC 优先建立 IPv6 端到端直连的能力。

**双端配置**：PC 与移动端均具备 IPv6 公网地址。

**验证结果**：

1. **IPv6 直连成功**：
   - `local-candidate` 和 `remote-candidate` 均为 `candidateType=host`，IP 地址均为公网 IPv6（`2403:...` 和 `2408:...`）
   - `candidate-pair` 状态为 `succeeded`，连接对由两个 host 类型候选者组成
   - 证明 WebRTC 成功协商并建立端到端 IPv6 直连通道，完全绕过 NAT 穿透和中继服务器

2. **IPv6 直连失败回退**（6.2.1 网络情况）：
   - `local-candidate` 为 `candidateType=host` 类型 IPv6 地址（`2403:ac00...`）
   - `remote-candidate` 为 `candidateType=srflx` 类型地址（`2408:8411...`）
   - 说明对端 IPv6 直连被运营商限制阻断，系统自动回退至 STUN 穿透

### 场景 9：浏览器兼容性测试

|--------|------|

### 场景 10：网络路径延迟对比测试

**目的**：对比三种 ICE 候选路径在真实网络环境下的传输延迟，量化验证 IPv6 降低延迟的理论分析。

**数据来源**：基于 `chrome://webrtc-internals` 中 `candidate-pair` 的 `currentRoundTripTime` 字段。

| 场景 | 候选类型 | currentRTT | 平均 RTT | 备注 |
|------|---------|-----------|---------|------|
| IPv6 直连 | host-host | 84ms | 84ms | - |
| IPv4 STUN 穿透 | host-srflx | 78ms | 88.7ms | - |
| TURN 中继 | relay-relay | 1ms | 1.3ms | 本地 coturn，不参考 |
