# 限制与实现细节：STUN 触发与映射工作流程

本文档说明在本项目（前端 `static/app.js` + 浏览器端 WebRTC）中，何时以及如何触发 STUN 请求、浏览器如何获得映射地址（server reflexive candidate，简称 `srflx`），以及相关的限制与验证方法。

## 概述

- 本项目的 `RTCPeerConnection` 创建时会传入 `iceServers`（当前代码默认使用 Google 的公开 STUN：`stun:stun.l.google.com:19302`）。
- 浏览器的 ICE agent 在添加本地轨道或创建 SDP（`createOffer` / `setLocalDescription`）时，会启动候选收集（gathering），在此过程中它会向配置的 STUN 服务器发送 STUN Binding Request 来探测本机在公网/本地网络上的“映射地址”。
- 当 STUN 服务器响应（Binding Response）后，浏览器形成 `srflx` 类型的 ICE candidate，并通过 `onicecandidate` 事件回调暴露给页面脚本。页面脚本可将该 candidate 发送给信令对端或用于本地显示/日志。

## 触发时机（简要序列）

1. 用户打开页面并点击“加入房间”：页面创建或重置 `RTCPeerConnection`（见 `createPeerConnection()`）。
2. 页面将本地媒体（通过 `getUserMedia` 或 `captureStream`）添加到 `PeerConnection`（`addTrack` / `addStream`），或准备在发送端发起协商（`maybeStartOffer()` 调用 `createOffer`）。
3. 浏览器的 ICE 收集器开始工作：它会为本地接口枚举 `host` 候选（本机/局域网地址），并并行地向配置的 STUN 服务器发起 STUN Binding Request 来获取 `srflx` 候选（公网映射），这一步在浏览器内部完成，网络路径可能通过 UDP（常见）或浏览器/系统提供的其他传输方式。
4. 每当浏览器生成一个新的候选（无论 `host`、`srflx` 还是 `relay`），都会触发 `peerConnection.onicecandidate` 回调，页面脚本会将 candidate 通过信令（本项目使用 WebSocket）发送给对端。
5. 在本项目中，`onicecandidate` 回调内的代码会调用 `parseIceCandidate()`（在 `static/app.js` 中），当检测到类型为 `srflx` 的候选时，会把 STUN 映射地址（`address:port`）记录到页面日志中（见 `STUN 映射成功！公网地址及端口` 的日志输出）。

## ICE restart（重新触发 STUN）

- 当发生网络短暂中断或连接状态变为 `disconnected` / `failed`，本项目会尝试触发 ICE restart：调用 `createOffer({ iceRestart: true })` 并设置本地描述，此时浏览器會为新的协商重新进行候选收集，包括向 STUN 服务器发起新的 Binding Request，从而获取可能已改变的 `srflx` 候选。
- 为避免频繁重启，本项目在前端实现了 30 秒冷却（`ICE_RESTART_COOLDOWN`），以降低短时网络波动导致的连续重启。

## 与 ngrok / 信令的关系

- ngrok 将 HTTP(S) 请求转发到本地的 WebSocket/HTTP 服务，但 STUN/TURN 流量不经过 ngrok（除非你显式把 TURN 服务器设在可被 ngrok 转发的某个主机并配置为通过 TCP/TLS 转发）。换言之，ngrok 不会改变浏览器与 STUN 服务器之间的直接通信流程。

## 常见限制与注意事项

- STUN 不是万金油：在对称 NAT（symmetric NAT）或受限运营商网络下，单纯 STUN 可能无法建立直连，需使用 TURN 中继流量。
- 浏览器可能出于隐私考虑限制本地地址泄露（最新浏览器在 candidate 隐私方面有策略变更），因此 `host` 候选的可见性会因浏览器/设置而异，但 `srflx`（经 STUN 获得的映射）通常仍可用。
- 链路本地 IPv6（`fe80::...`）需要带接口索引（zone id，如 `%12`），在浏览器显示/访问时可能受限或显示格式不同；某些环境下浏览器可能不支持带 zone 的地址直接作为候选。
- STUN 使用 UDP（多数情况），如果网络阻止 UDP 出站，浏览器可能无法和 STUN 通信，此时若没有 TURN，会导致候选收集失败。

## 调试与验证方法

1. 在浏览器控制台查看 `onicecandidate` 触发的 candidate 字符串，确认是否存在 `typ srflx` 条目及其地址/端口。
2. 在本项目页面观察日志（页面右下日志区）是否出现类似 `STUN 映射成功！公网地址及端口: x.x.x.x:yyyy` 的条目；这表示浏览器已从 STUN 收到映射并生成 `srflx` 候选。
3. 若需要复现或自动化验证，可在浏览器端注入 fake STUN/TURN 或在受控网络（例如开启/关闭 UDP 出站）下观察候选收集差异。

## 建议

- 若课堂或测试场景要求在受限网络下仍能保证连通，建议同时配置 TURN 服务器 并将其加入 `iceServers` 配置；TURN 能用作中继，保证在无法直接 P2P 的场景下媒体流仍能传输。
- 在对外演示时优先使用 HTTPS（ngrok 的 `https://`）并在需要的情况下临时配置简单鉴权（如房间密码），减少滥用或误连风险。

问：那为什么外面访问我手机热点的ipv6公网也访问不了，是运营商限制吗
答：是的，运营商限制是最常见原因之一，而且通常不止一个因素叠加。

你这个现象“手机热点拿到 IPv6 公网地址，但外网还是访问不了”一般由下面几类原因导致：

1. 运营商默认拦截入站连接  
很多移动网络会给你分配全局 IPv6，但只允许出站，不允许外部主动打进来（相当于运营商侧防火墙）。

2. 手机系统本身有入站防护  
热点模式下，手机对共享出来的终端常常有额外防护策略，未建立会话的外部连接会被丢弃。

3. 你电脑未真正监听在 IPv6 上  
如果服务只监听 IPv4，就算有 IPv6 地址也无法访问。  
你当前代码理论上会尝试双栈，但要以实际启动日志为准（是否出现 :: 监听成功）。

4. 本机防火墙拦截  
Windows 防火墙可能允许本地访问，不允许来自外网的 IPv6 入站 TCP 8081。

5. 地址可达性不稳定  
移动网 IPv6 地址可能频繁变化，或者是临时地址；你复制出去后很快失效。

你可以按这个顺序快速排查：

1. 确认服务启动后日志里有 IPv6 监听成功（例如绑定到 ::）。  
2. 在同一台电脑本机先测试 IPv6 回环访问。  
3. 放行 Windows 防火墙的 8081 入站（IPv4/IPv6 都放行）。  
4. 用外部网络设备测试访问，而不是同一热点内测试。  
5. 若仍不通，基本可判定是运营商/移动网络入站策略限制。

结论上，你可以在报告里这样写：  
“移动网络虽提供 IPv6 地址，但受运营商入站访问策略与终端防火墙影响，公网设备无法直接访问热点下主机服务，该限制属于网络环境约束而非应用实现缺陷。”

3.x 网络可达性限制与环境约束分析

尽管本实验在协议层面完整实现了 WebRTC 与 IPv6 双栈支持，但在实际校园网与移动网络环境下，仍存在网络边界策略导致的可达性限制，这属于基础设施层面的约束，而非应用实现缺陷。

（1）校园网环境下的访问范围

在校园网中，系统可稳定支持同一网段或校内策略允许范围内的设备互联，但无法实现跨公网的普遍可达，主要原因包括：

入站连接受限：校园网出口防火墙通常禁止外部设备主动发起入站连接。

IPv6 地址 ≠ 公网可达性：即便主机获取到 IPv6 全局地址，也不代表外网可自由访问。

信令与媒体通道耦合受限：若 HTTP/WebSocket 信令页面无法被外部访问，WebRTC 媒体协商与传输便无从启动。

因此，本实验将验证范围限定为局域网与校内可控环境，重点验证 WebRTC 协议流程与双栈行为。

（2）移动热点 IPv6 访问限制

在移动网络（手机热点）场景下，尽管终端可分配到 IPv6 公网地址，但仍普遍存在外网不可达现象，主要原因为：

运营商级入站拦截：多数移动网络默认禁止外部主动访问热点下主机。

终端系统防护策略：热点共享模式下，手机系统会对入站流量施加额外过滤。

地址动态性与临时地址：IPv6 地址可能频繁变化，导致外部访问失效。

主机防火墙规则：操作系统防火墙未显式放行对应端口的 IPv6 入站流量。

（3）结论

综上，本实验在功能实现与协议支持层面是完整的，而跨公网访问受限是由运营商策略、校园网防火墙与终端防护共同导致的环境约束。在课程实验与协议分析中，该限制不影响 WebRTC 核心机制的验证与评估。
