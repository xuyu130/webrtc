# WebRTC 协议分析与基于 WebRTC 的视频传输实现

## 1. 简介

WebRTC 是一种面向实时音视频通信的浏览器原生技术栈。它的核心价值不是单一的数据传输，而是把信令、NAT 穿透、传输安全、媒体编解码和实时媒体流控制整合成一个可直接落地的方案。

本项目实现了一个最小但完整的 WebRTC 视频传输原型：Python 提供信令服务和静态网页，浏览器作为发送端和接收端完成点对点视频协商与传输。该方案可运行、可自测，并能直观展示 WebRTC 建链过程。

## 2. 基本原理

WebRTC 的典型流程包括：

1. 信令交换：双方通过 WebSocket、HTTP 或其他方式交换 SDP 和 ICE 候选地址。
2. 会话协商：发送端创建 offer，接收端创建 answer，双方根据 SDP 确认媒体能力与传输参数。
3. ICE 建链：双方通过 ICE/STUN 探测可达路径，必要时可借助 TURN 中继。
4. 传输加密：媒体内容通常通过 DTLS-SRTP 保护，保证链路安全。
5. 媒体流播放：接收端将远端轨道挂载到 video 元素，完成实时显示。

我对 WebRTC 的理解是：它真正解决的不是“传输视频”这件事本身，而是“如何在复杂网络环境下以尽可能低的接入成本建立安全、实时、低时延的媒体通道”。这也是它比传统客户端-服务器推流更适合浏览器实时场景的原因。

## 3. 设计与实现

### 3.1 系统结构

本项目分为三部分：

- Python 信令服务器：负责页面托管和 WebSocket 消息转发
- 发送端浏览器：采集摄像头或画布模拟流，创建 offer
- 接收端浏览器：创建 answer，接收远端轨道并显示

### 3.2 核心实现

信令服务器采用房间机制。房间内最多包含一个发送端和一个接收端，双方通过 JSON 消息交换 offer、answer 和 candidate。这样设计的优点是逻辑简单、便于调试，也符合课程演示场景。

前端实现中，发送端会优先调用 `getUserMedia` 获取摄像头；如果权限不可用，则自动切换为画布生成的模拟视频流，保证实验仍可运行。接收端在收到 offer 后完成 SDP 协商并播放远端画面。

### 3.3 本项目的特色

- 支持摄像头与画布模拟流双模式，增强了可运行性
- 页面显示连接状态、ICE 状态和统计信息，便于演示协议过程
- 使用房间机制支持多次实验，便于录屏和结果复现

## 4. 结果展示与分析

### 4.1 结果展示

实验现象如下：

- 发送端页面显示本地视频或模拟画面
- 接收端页面在协商完成后显示远端视频
- 日志区记录了连接建立、对端加入、ICE 状态变化等信息

### 4.2 分析

从实验结果可以看出，WebRTC 的建链过程具有明显的阶段性：先是信令可达，再是 SDP 协商，最后是 ICE 打通可用链路。也就是说，WebRTC 的“实时性”并不意味着过程简单，反而需要对网络可达性和媒体协商有更细粒度的控制。

我的结论是：如果课程项目强调“新兴网络技术的运行原理”，WebRTC 是一个很好的选题。它既能展示协议分析，又能通过一个可运行原型体现工程实现能力，而且非常适合做延迟、丢包、分辨率变化等实验。

## 5. 不足与改进方向

当前版本是一个最小实现，主要不足在于：

- 默认以点对点演示为主，没有加入 TURN 中继实验
- 没有实现多方会议或复杂媒体路由
- 统计信息还可以继续扩展，例如码率、丢包率和端到端时延

后续可以加入：

- STUN/TURN 对比实验
- 网络限速与丢包模拟
- 多用户房间和媒体转发
- 编码参数自适应实验

## 6. 结论

本项目实现了一个可以直接运行的 WebRTC 视频传输演示系统，并以此为基础完成协议分析框架。它的价值在于把抽象的 WebRTC 流程具体化，让“信令、协商、ICE、传输”这些概念能够通过真实页面和日志观察到。
