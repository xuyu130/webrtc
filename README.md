# WebRTC 协议分析与视频传输实验

这是一个可直接运行的 WebRTC 作业原型：Python 负责提供静态页面和 WebSocket 信令，浏览器负责采集摄像头并建立 WebRTC 连接，最终完成实时视频传输。

## 目录说明

- `app.py`：Python 信令服务与静态资源服务
- `static/index.html`：演示页面
- `static/app.js`：WebRTC 建链、信令交换和统计信息
- `static/style.css`：页面样式
- `report.md`：可直接用于课程报告的内容草稿

## 运行方式

1. 安装依赖：`pip install -r requirements.txt`
2. 启动服务：`python app.py`
3. 打开浏览器访问启动输出中打印的可访问地址；默认情况下程序会同时监听本机回环、局域网 IPv4 和 IPv6 地址，本机通常可用 `http://127.0.0.1:8080`，局域网访问则使用启动时打印的 `http://<局域网IP>:8080` 或 `http://[IPv6]:8080`
4. 打开两个标签页，一个选“发送端”，另一个选“接收端”
5. 两边加入同一房间，发送端点击“开始推流”

## 改端口与手机访问

如果你想让手机访问，需要让服务监听到局域网地址，而不是只监听本机回环地址。可以这样启动：

1. 监听所有网卡并改端口，例如：`HOST=0.0.0.0 PORT=8081 python app.py` `$env:HOST='0.0.0.0'; $env:PORT='8081'; python app.py`
2. 也可以显式监听 IPv6，例如：`HOST=:: PORT=8081 python app.py`
3. 启动后会自动打印本机检测到的可访问地址，包括局域网 IPv4 和 IPv6，优先使用这些输出中的 URL
4. 手机上连接同一个 Wi-Fi，然后访问对应的局域网地址；如果你是通过手机热点互通，通常直接用热点分配给电脑的地址访问即可
5. 如果端口改成了别的值，手机访问地址也要同步改成对应端口
```
无线局域网适配器 WLAN 2:

   连接特定的 DNS 后缀 . . . . . . . :
   IPv4 地址 . . . . . . . . . . . . : 172.20.10.3
   子网掩码  . . . . . . . . . . . . : 255.255.255.240
   默认网关. . . . . . . . . . . . . : 172.20.10.1
```
修改目的：让页面既可以本机访问，也可以在同一局域网内通过手机打开。

## 在手机上访问与摄像头权限说明

浏览器要访问手机摄像头通常要求页面在“安全上下文”（HTTPS）下。实践中：

- Android（Chrome）在同一局域网通过 IP 访问时常允许 `getUserMedia`，但并不保证所有设备/版本都可用。
- iOS（Safari）对 HTTPS 的要求更严格，若出现无法获取摄像头的情况，请使用下列替代方案：

   1. 使用 `ngrok` / `localtunnel` 将本地服务映射为 HTTPS，例如：
   ```bash
   ngrok http 8081
   ```

   2. 在开发机为 `aiohttp` 提供 SSL（自签名证书），并以 `HOST=0.0.0.0 PORT=8081 python app.py` 启动，然后通过 `https://<ip>:8081` 访问（需在手机上信任证书或接受警告）。

   3. 或者把页面部署到支持 HTTPS 的服务器上（推荐生产环境）。

本项目的前端已增强移动端支持：页面会尝试按设备 ID 或 `facingMode` 选择后置/前置摄像头，并在检测到多个摄像头时显示“切换摄像头”按钮。若仍无法使用摄像头，请按上面的 HTTPS 建议排查。

补充说明：如果程序是在 WSL 里启动的，Windows 浏览器通常优先用 `http://127.0.0.1:8081` 访问；手机要直连局域网访问时，通常需要在 Windows 主机环境启动服务，或者额外配置端口转发。

## IPv6 支持、双栈与测试

IPv6 过渡技术里，“双栈”指的是同一台主机、同一个服务同时支持 IPv4 和 IPv6。对外表现为：客户端既可以通过 IPv4 地址访问，也可以通过 IPv6 地址访问。它的核心目标不是“把一种协议翻译成另一种协议”，而是让两种协议并存，从而降低迁移成本。

本项目现在实现的就是这一类“双栈”方式：`app.py` 默认会同时尝试绑定 `0.0.0.0` 和 `::`，所以同一端口既能被 `http://127.0.0.1:8080` 或 `http://172.x.x.x:8080` 访问，也能被 `http://[IPv6]:8080` 访问。启动时打印的可访问 URL 也是为了让你直接拿到本机当前可用的 IPv4 / IPv6 地址。

这和“翻译”或“隧道”不同：

- 双栈：IPv4 和 IPv6 都在本机原生运行，应用不需要知道对方是哪个协议族；
- 翻译：例如 NAT64 / 464XLAT，把 IPv6 流量转换成 IPv4，或者反过来做地址/报文层转换；
- 隧道：例如 IPv6-over-IPv4，把 IPv6 报文封装在 IPv4 里穿过只支持 IPv4 的链路。

如果你的课程题目要求“分析并模拟实现 IPv6 过渡技术”，那当前代码对应的是“双栈”这一路线，适合用来演示“主机如何同时面向 IPv4 与 IPv6 提供服务”。如果老师希望看到翻译或隧道，还需要另外补充对应的模拟实验。

### 如何验证双栈

1. 启动服务时，优先观察控制台输出。如果看到类似 `Bound addresses: 0.0.0.0, ::`，就说明当前服务同时接受 IPv4 和 IPv6 连接。
2. 在 Windows 上可用 `netstat -ano | findstr 8080` 或 `Get-NetTCPConnection -LocalPort 8080` 查看监听地址；看到 `0.0.0.0:8080` 和 `:::8080` 说明双栈生效。
3. 在本机用 `http://127.0.0.1:8080` 测试 IPv4 回环访问，在另一台支持 IPv6 的设备上用 `http://[你的IPv6地址]:8080` 测试 IPv6 访问。
4. 如果局域网设备打不开，通常优先检查防火墙、网络隔离策略和是否拿到了真正可路由的 IPv6 地址。

### 如果要专门做 IPv6 环境测试

1. 只想看 IPv6 地址时，可以显式启动：
```powershell
$env:HOST='::'; $env:PORT='8081'; python app.py
```
在 Linux / WSL / macOS 下类似：
```bash
HOST='::' PORT=8081 python app.py
```

2. 在主机上查看 IPv6 地址：
- Windows：运行 `ipconfig /all`，查找活动适配器下的 `IPv6 地址`。
- Linux / WSL：运行 `ip -6 addr show` 或 `hostname -I`。

3. 在另一台设备上访问：
- 使用全局或站点内唯一的 IPv6 地址更可靠，例如 `http://[2001:db8::1234]:8081`。
- 若只能获得链路本地地址 `fe80::...`，访问时要带接口索引，例如 `http://[fe80::abcd%12]:8081`，但部分浏览器可能不支持。

4. 常见注意事项：
- Windows 防火墙需要放行对应端口。
- 校园网或 Wi‑Fi 可能限制客户端互访，导致虽然有 IPv6 地址，但设备之间仍然不通。
- iOS Safari 对 HTTP 和自签名证书更敏感，必要时要配合 HTTPS。

### 小结

如果你要在报告里写“IPv6 过渡技术分析及其模拟实现”，当前项目最准确的表述是：本实验通过一个 WebRTC 信令服务演示了双栈接入，即同一服务同时支持 IPv4 和 IPv6 访问；它属于过渡技术中的双栈实现，而不是翻译或隧道实现。

## 远程访问与 ngrok / WSS（注意）

- 使用 `ngrok` 等反向代理将本地服务暴露到公网时，请通过 ngrok 提供的 `https://` 地址在远程设备（例如手机、同学的电脑）上访问。前端会根据 `location.protocol` 自动选择 `ws:` 或 `wss:`，因此在 HTTPS 页面上信令会自动使用 `wss`（无需修改前端）。
- ngrok 将 HTTP(S) 请求转发到本地端口（例如 `http://localhost:8081`），而浏览器中的 WebRTC 媒体协商（STUN/TURN）是客户端（浏览器）直接与 STUN/TURN 服务器通信的，ngrok 不改变 STUN 的工作流程。
- 注意事项：
   - iOS/Safari 强制要求 HTTPS 才能访问摄像头/麦克风，请始终使用 ngrok 的 `https://` 地址进行移动端测试。
   - 免费 ngrok 会话的域名通常短期有效；如需长期稳定的公网地址，请考虑付费保留域名或自建反向代理。
   - 经由公网测试时，点对点连接可能受 NAT 类型影响；若测试中发现无法建立媒体通道，请考虑配置 TURN 服务器 来保证连通性。
   - 暴露本地服务存在安全风险，避免在公开演示中泄露敏感数据；可临时加上简单房间密码或在信令层增加最小鉴权。
